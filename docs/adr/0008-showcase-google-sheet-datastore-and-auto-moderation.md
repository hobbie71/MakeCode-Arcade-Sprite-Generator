# The Showcase stores Submissions in a Google Sheet (via Apps Script) and auto-publishes them with AI moderation, not a human gate

The app is adding a **Showcase**: a home-page grid of community **Submissions** (a maker name + title + sprite, plus the private prompt), where makers Submit straight from the studio's Export flow. The maintainer is self-funding the project and explicitly does not want a paid database or a backend that costs money to run idle. These constraints — $0 forever, no real DB, manual effort minimised, but a *live* wall that updates as a classroom submits — drove two non-obvious choices that the rest of the feature is built on. The terms (Showcase, Submission, Submit, Examples, Maker) are defined in `CONTEXT.md`; the semantics were settled in a one-question-at-a-time design interview (2026-06-12), written up in `docs/superpowers/specs/2026-06-12-community-showcase-design.md`.

## Decisions

1. **A Google Sheet is the datastore, reached through a Google Apps Script web app — not a database.** A ~25-line Apps Script bound to the sheet exposes `doPost` (append a Submission) and `doGet` (return the current rows). It is free, unlimited at classroom scale, needs no API keys or billing account, and — critically — the sheet itself is the maintainer's admin UI (delete a row to remove a Submission). Rejected: Supabase/Postgres (a real DB, free tier pauses on inactivity, and the maintainer specifically didn't want one), Upstash Redis (no spreadsheet-style review surface), and a committed static JSON (Render's filesystem is ephemeral, so the server can't durably write to the repo).

2. **The Apps Script is reached only by the server, never the browser.** Both writes and reads proxy through the Hono server. `doPost` is guarded by a secret token only the server holds; the server also holds the script URL. This (a) sidesteps the notoriously flaky browser-to-Apps-Script CORS behaviour, (b) keeps the URL and token out of the public bundle, and (c) lets the server cache reads (~15s TTL) so any number of open "live wall" tabs collapse to ~4 Google reads/minute — making the free Apps Script quota un-blowable regardless of traffic. Cost accepted: the home-page grid now depends on the Render service being awake; a `localStorage` stale-while-revalidate cache plus an honest empty state keep a cold start from ever looking broken.

3. **Submissions auto-publish instantly after AI moderation; there is no human approval step.** The maker's experience is "submit → it's on the site, right there and then." OpenAI's `omni-moderation` (free, already used for prompts, and image-capable) screens the **name + title** (text) and the **rendered sprite image** at Submit time, with kid-stricter thresholds layered on via the existing `moderation-logic.ts` override pattern. Rejected: a human review gate (kills the live-wall magic the feature exists for) and a hybrid per-subset gate (re-introduces the latency for exactly the content makers are proudest of). The page is minors-facing and public, so this is a deliberate safety trade-off, made tolerable by three backstops below.

4. **Any sprite is eligible, and a small rolling window bounds exposure.** Generated, uploaded, pasted, and hand-drawn sprites can all be submitted — excluding hand-drawn work would cut the heart out of the feature, and post-generation editing means even AI sprites must be image-moderated anyway, so restricting by origin buys little. The sheet holds only the ~18 most recent Submissions (12 shown + a ~6 buffer for deletions); older rows roll off automatically.

## Consequences

- **Residual moderation risk is accepted, not eliminated.** `omni-moderation` is trained on photographs, not 16×16 pixel art, and there is no OCR — a crude shape or a slur spelled in pixels can slip through. The backstops are: image moderation on every Submission (the rendered sprite is upscaled before sending to give the model more signal), the rolling window (nothing lingers), and one-click deletion from the sheet (the maintainer watches the classroom live). The feature should not be presented as "guaranteed safe."
- **The Showcase is full-stack**, touching `shared/` (new wire schemas), `server/` (a new route, a moderation extension, an Apps Script client + read cache, config), and `client/` (a Submit modal in the studio + the home-page section). This is a different scope from the client-only motion redesign, even though both land on `HeroPage`.
- **One-time manual setup is required of the maintainer**: create the sheet, paste + deploy the Apps Script, and put the resulting URL + token in the server's env (`APPS_SCRIPT_URL`, `APPS_SCRIPT_TOKEN`). The feature is inert until that exists.
- **Liveness is eventually-consistent, bounded by the read cache + poll interval** (~15s each), not instant for *other* viewers; the submitter sees their own sprite immediately via optimistic insert. This is acceptable for a classroom wall and avoids the complexity/fragility of SSE on Render's free tier.
- **Submit is best-effort and fully decoupled from Copy.** A moderation rejection, rate-limit, or Apps Script outage never blocks or undoes the clipboard Copy that triggered the modal — the sprite is always still copied.
